By removing CommonJS from published code, we’ve simplified Storybook and reduced its install size by 29% while simultaneously un-minifying our distribution for easier debugging.</p><p>Storybook 10 also includes key improvements to level up your UI development, documentation, and testing workflows:</p><ul><li>🧩&nbsp;Module automocking for easier testing</li><li>🏭&nbsp;Typesafe CSF factories Preview for React</li><li>💫&nbsp;UI editing and sharing optimizations</li><li>🏷️&nbsp;Tag filtering exclusion for sidebar management</li><li>🔀&nbsp;Svelte async components, Next 16, Vitest 4, and more</li></ul><p>And finally, while you’re here, we’re also excited to share some experimental work for feedback:</p><ul><li>🧪&nbsp;Test syntax for familiarity and reduced sidebar clutter</li><li>⚛️&nbsp;Component testing RSCs</li></ul><h2 id="esm-only">ESM-only</h2><p>ESM-only is the main breaking change in Storybook 10. It reduces install size by 29% on top of the 50% savings we released in <a href="https://storybook.js.org/blog/storybook-9/?ref=storybookblog.ghost.io">Storybook 9</a>.</p><p>More importantly, it helps push the Javascript community forward towards a cleaner, simpler future. The ecosystem is healing. We are so excited about this change, we think it deserves <a href="https://storybook.js.org/blog/storybook-is-going-esm-only/?ref=storybookblog.ghost.io">its own post</a>.</p><p>ESM-only requires a modern Node that supports <code>ESM require</code> (<code>20.16+</code>, <code>22.19+</code>, or <code>24+</code>).</p><figure class="kg-card kg-image-card"><img alt="Storybook 10" class="kg-image" height="162" src="https://storage.ghost.io/c/f9/d7/f9d78a80-56f8-4adc-8ba1-0e2d6ee72b16/content/images/2025/11/Storybook-10-Image.png" width="1061" /></figure><h2 id="module-automocking">Module automocking</h2><p>In <a href="https://storybook.js.org/blog/storybook-9/?ref=storybookblog.ghost.io">Storybook 9</a>, we teamed up with Vitest to create the best way to test UI components. Users tell us the combination of Storybook’s interactive UI and Vitest’s blazing fast test runner is a match made in heaven.</p><p>But they had less kind things to say about our approach to module mocking, which apparently came from another place. 👿&nbsp;Undeterred, we went back to the drawing board and collaborated with Vitest to create a next-generation module mocking solution.</p><p>Storybook’s new <code>sb.mock</code> is inspired by <code>vi.mock</code>, but it’s simpler, compatible with both Vite and Webpack builders, and available in both development and static production builds. Read all about it in our <a href="https://storybook.js.org/blog/next-generation-module-mocking/?ref=storybookblog.ghost.io">feature announcement post.</a></p><figure class="kg-card kg-image-card"><img alt="Storybook 10" class="kg-image" height="983" src="https://storage.ghost.io/c/f9/d7/f9d78a80-56f8-4adc-8ba1-0e2d6ee72b16/content/images/2025/11/Storybook-10-image--1--1.png" width="2000" /></figure><h2 id="typesafe-csf-factories">Typesafe CSF Factories</h2><p>Component Story Format (CSF) is how to write component examples and tests in Storybook. We created it in 2019, back when JavaScript ruled the roost and <a href="https://flow.org/?ref=storybookblog.ghost.io">Flow</a> was still a thing. Fast forward to today, and TypeScript is the norm, making up 80+% of Storybook projects.</p><p>To keep up, we’re introducing <a href="https://storybook.js.org/docs/10/api/csf/csf-next?ref=storybookblog.ghost.io">CSF Factories</a>, the next major iteration of CSF. Factories provide better type safety, ergonomics, and autocompletion.</p><pre><code class="language-ts">// CSF 3
import type { Meta, StoryObj } from '@storybook/nextjs-vite';
import Button from './Button';

const meta = { component: Button } satisfies Meta&lt;typeof Button&gt;;
export default meta;

type Story = StoryObj&lt;typeof meta&gt;;

export const Primary: Story = {
  args: { label: 'Button', primary: true }
};

// CSF Factories - Less boilerplate and no type assignments! 🎉
import preview from '../.storybook/preview';
import Button from './Button';

const meta = preview.meta({ component: Button });

export const Primary = meta.story({
  args: { label: 'Button', primary: true }
});
</code></pre><p>With Storybook 10 we are promoting CSF Factories from <code>Experimental</code> to <code>Preview</code> status, which means we don’t anticipate any significant changes when we make Factories the default in Storybook 11 next Spring. There are codemods to automatically upgrade your existing stories, but you are not obligated to upgrade. Older versions of CSF are still supported and will be for the foreseeable future.</p><p>CSF Factories are available for React only, and we expect to release Vue, Angular, and Web Components factory support in 10.x.</p><h2 id="ui-editing-and-sharing-optimizations">UI editing and sharing optimizations</h2><p>At Storybook, we sweat the details, continuously refining our UI to make you more productive. Storybook 10 includes another round of optimizations for easier story editing and sharing.</p><p>Want to see how that component looks on your mobile phone? The sharing menu now has a QR code for easy access:</p><figure class="kg-card kg-image-card"><img alt="Storybook 10" class="kg-image" height="441" src="https://storage.ghost.io/c/f9/d7/f9d78a80-56f8-4adc-8ba1-0e2d6ee72b16/content/images/2025/11/Screenshot-2025-10-14-at-9.26.29---PM-1.png" width="450" /></figure><p>Want to edit that failing story in your favorite editor? That’s now a click away too:</p><figure class="kg-card kg-image-card"><img alt="Storybook 10" class="kg-image" height="520" src="https://storage.ghost.io/c/f9/d7/f9d78a80-56f8-4adc-8ba1-0e2d6ee72b16/content/images/2025/11/Screenshot-2025-10-14-at-9.29.56---PM.png" width="618" /></figure><p>Got suggestions for more UI improvements? Feature requests are <a href="https://github.com/storybookjs/storybook/discussions?ref=storybookblog.ghost.io" rel="noopener noreferrer">always welcome</a>.</p><h2 id="tag-filtering-exclusion-and-configuration">Tag filtering exclusion and configuration</h2><p>Storybook CSF makes it easy to capture every state of your UI. But stories can pile up quickly, with the largest Storybooks containing thousands of stories spanning use cases of both documentation and testing, capturing components in different stages of maturity, owned by different teams, and so on.</p><p>To make large Storybooks easier to manage, we added tags to stories:</p><pre><code class="language-ts">// Button.stories.ts

export const GlassMode: Story = {
  args: { glass: true },
  tags: ['experimental'],
};
</code></pre><p>In Storybook 8.5, we added a filtering UI to filter the the sidebar down to tags matching a particular tag. For example, if you wanted to only show stories with the <code>experimental</code> tag, you could select the <code>experimental</code> tag in the UI:</p><figure class="kg-card kg-image-card"><img alt="Storybook 10" class="kg-image" height="420" src="https://storage.ghost.io/c/f9/d7/f9d78a80-56f8-4adc-8ba1-0e2d6ee72b16/content/images/2025/11/Screenshot-2025-10-14-at-9.35.24---PM.png" width="734" /></figure><p>A good start, but users immediately wanted more. So in Storybook 10 we’ve beefed up tags with two new features: exclusion and configuration.</p><p>Exclusion makes it possible to exclude stories based on selected tags. So if you want to hide stories with the <code>experimental</code> tag, you can do that too:</p><figure class="kg-card kg-image-card"><img alt="Storybook 10" class="kg-image" height="418" src="https://storage.ghost.io/c/f9/d7/f9d78a80-56f8-4adc-8ba1-0e2d6ee72b16/content/images/2025/11/Screenshot-2025-10-14-at-9.36.40---PM.png" width="706" /></figure><p>Not only that, but you can configure the default UI state in the configuration file.</p><pre><code class="language-ts">// .storybook/main.ts

const config: StorybookConfig = {
  tags: {
    experimental: { defaultFilterSelection: 'exclude' },
  },
};
</code></pre><p>To learn all about Storybook tags, please see <a href="https://storybook.js.org/docs/writing-stories/tags?ref=storybookblog.ghost.io">the documentation</a>.</p><h2 id="svelte-async-components-and-state-mocking">Svelte async components and state mocking</h2><p>Thanks to our vibrant collaboration with the Svelte community, Storybook for Svelte keeps getting better. In Storybook 10, we’ve added support for <a href="https://github.com/storybookjs/storybook/pull/31476?ref=storybookblog.ghost.io">async components</a>, Svelte’s latest innovation. We’ve also added <a href="https://www.notion.so/1e56e81620348051bf7ce6271919324f?pvs=21&amp;ref=storybookblog.ghost.io">mocking support for<code>app/state</code></a>, closing a long-standing gap in our SvelteKit integration. If you’re a Svelte user and haven’t tried Storybook in awhile, we highly recommend giving it another try!</p><h2 id="next-16-vitest-4-and-more">Next 16, Vitest 4, and more!</h2><p>Every Storybook release is a chance to upgrade with ecosystem. Storybook 10 adds support for Next 16 and Vitest 4, two of the biggest releases this Fall. We've done this while preserving support for older versions.</p><h2 id="try-storybook-10-today">Try Storybook 10 today</h2><p>Try Storybook 10 in a new project:</p><pre><code class="language-bash">npm create storybook@latest
</code></pre><p>In an existing Storybook project, use our automated migration wizard to help you upgrade:</p><pre><code class="language-bash">npx storybook@latest upgrade
</code></pre><p>We also provide a&nbsp;<a href="https://storybook.js.org/docs/9/migration-guide?ref=storybookblog.ghost.io">migration guide</a>&nbsp;to help fill in the gaps.</p><h2 id="what%E2%80%99s-next-test-syntax-rsc-testing">What’s next? Test syntax, RSC testing</h2><p>Every release we announce what’s next and point at our roadmap. This time we’re doing things a little different and we have a bunch of stuff cooking that you can try today and help us shape the future of UI development together.</p><h3 id="experimental-test-syntax">Experimental test syntax</h3><p>Building on the CSF Factories syntax and Tag exclusion filtering exclusion described above, we’ve made it possible to write tests in a familiar syntax and then easily exclude them from the sidebar to remove clutter for non-technical collaborators:</p><pre><code class="language-ts">// Button.stories.ts
import preview from '../.storybook/preview';

import { Button } from './Button';

const meta = preview.meta({ component: Button });

export const Disabled = meta.story({ args: { disabled: true } });

// 🆕 .test method: Attach tests to a story!
//    The test function can run the same code as the play function
Disabled.test('should be disabled', async ({ canvas, userEvent, args }) =&gt; {
  const button = await canvas.findByRole('button');
  await userEvent.click(button);

  await expect(button).toBeDisabled();
  await expect(args.onClick).not.toHaveBeenCalled();
});
</code></pre><p>For more information, see our <a href="https://github.com/storybookjs/storybook/discussions/30119?ref=storybookblog.ghost.io">RFC</a> and join our early access program to discuss potential improvements.</p><h3 id="experimental-rsc-component-testing">Experimental RSC component testing</h3><p>Finally, we are collaborating with the entire React ecosystem (React core, Next, Testing Library, Vite, and Vitest) to solve component testing for React Server Components (RSCs).</p><p>Until now, RSCs have been exclusively tested end-to-end (E2E), since they span both the client and server. However, it is challenging to scale E2E to a large number of tests due to performance and flake. Comparatively, component tests are fast, highly controlled, and low-flake.</p><figure class="kg-card kg-image-card"><img alt="Storybook 10" class="kg-image" height="106" src="https://storage.ghost.io/c/f9/d7/f9d78a80-56f8-4adc-8ba1-0e2d6ee72b16/content/images/2025/11/image--6-.png" width="200" /></figure><p>We’ve released experimental RSC component testing for Storybook and have opened an early access program to stabilize it with users. If you don’t use Storybook, and want to component test your RSCs directly in Vitest, we’ve also got you covered. Read more about how it in our <a href="https://github.com/storybookjs/storybook/discussions/32665?ref=storybookblog.ghost.io">RFC</a>.</p><h2 id="credits">Credits</h2><h3 id="core-team">Core
