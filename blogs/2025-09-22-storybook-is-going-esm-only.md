<img alt="Storybook is going ESM-only" src="https://storage.ghost.io/c/f9/d7/f9d78a80-56f8-4adc-8ba1-0e2d6ee72b16/content/images/2025/09/Storybook-ESM-Only.jpg" /><p>ESM has been "the future" of JavaScript for a long time. Unfortunately, for just as long, it's been in transition.</p><p>Some tools adopted ESM early, pushing the ecosystem forward. Others hesitated, because migration is difficult. Many libraries didn’t switch to ESM because it would be a breaking change. Those that braved the switch to push the ecosystem forward risked creating struggle for users and maintainers, with frustrating upgrade paths and complex compilation setups.</p><p>Now Storybook is ready to do our part in embracing this standard. <strong>We believe ESM is the future</strong> and we're finally able to take the leap ourselves.</p><h2 id="%F0%9F%A7%AD-why-is-esm-a-big-deal">🧭&nbsp;Why is ESM a big deal?</h2><p>ESM is the <strong>official</strong>, <strong>standardized</strong> module system for JavaScript. It works natively in both browsers and Node.js, and forms the foundation of modern runtimes like Deno and Bun. </p><p>By contrast, CommonJS is the non-standardized module system adopted by much of the ecosystem prior to ESM being widely available.</p><p>The <strong>entire</strong> JavaScript ecosystem is converging on the ESM standard.</p><p>Managing two parallel module systems (ESM and CJS) creates significant challenges. Library authors face increased complexity, while users encounter frustrating and hard to debug issues.</p><p>For years, the Storybook team wanted to advance the ecosystem (and reap the performance benefits) by also adopting the ESM-only approach.</p><p>Unfortunately, we had to keep postponing this transition. The potential for disruption was considerable, and we lacked a clear path forward that wouldn't break substantial portions of the ecosystem.</p><h2 id="%F0%9F%AB%A4-dual-builds-were-painful">🫤&nbsp;Dual builds were painful</h2><p>To support both CommonJS and ESM, Storybook has been shipping dual builds. We compiled and distributed our source code twice: both as ESM and CommonJS.</p><p>This approach worked for a while, but came with significant drawbacks:</p><ul><li>Nearly doubled package sizes</li><li>Greater maintenance complexity</li><li>Subtle bugs from duplicate or mismatched packages</li><li>Necessary workarounds in builders and core to prevent these bugs</li></ul><p>For Storybook 10, we were planning to finally make the transition to ESM-only and accept the disruption.</p><p>However, thanks to the incredible work on <code>require(esm)</code> by <a href="https://github.com/JoyeeCheung?ref=storybookblog.ghost.io">Joyee Cheung</a> (and everyone else involved), we can now make this transition for Storybook without breaking the ecosystem around it.</p><p>To everyone involved in that work, we cannot thank you enough for making that happen. If you have not read her <a href="https://joyeecheung.github.io/blog/2024/03/18/require-esm-in-node-js/?ref=storybookblog.ghost.io">blog post</a> or seen her amazing <a href="https://www.youtube.com/watch?v=YRueCer2kig&amp;ref=storybookblog.ghost.io">talk</a> about the matter, we highly recommend both.</p><h2 id="%F0%9F%AA%B6-dual-builds-caused-bloat">🪶 Dual builds caused bloat</h2><p>Dual builds added extra weight for everyone installing and using Storybook.</p><p>Switching to ESM-only reduces the install size by ~<strong>15%</strong> compared to Storybook 9.</p><p>That reduction could have been closer to <strong>50%.</strong> After all, we did stop shipping one of the two distributions. The Storybook team chose to trade a bigger headline number for publishing <strong>readable, unminified code</strong>.</p><p>This has important advantages:</p><ul><li>The code is human-readable out of the box</li><li>It’s easier to debug and maintain</li><li>It can be inspected and audited for security issues</li></ul><p>We think a 15% size reduction is nothing to scoff at, and shipping unminified code will make Storybook a better part of the ecosystem, long term.</p><figure class="kg-card kg-image-card kg-width-wide"><img alt="Storybook is going ESM-only" class="kg-image" height="1466" src="https://storage.ghost.io/c/f9/d7/f9d78a80-56f8-4adc-8ba1-0e2d6ee72b16/content/images/2025/09/Keyframe-Storybook-ESM.png" width="2000" /></figure><h2 id="%F0%9F%A7%B1what%E2%80%99s-changing-in-storybook-10">🧱What’s changing in Storybook 10?</h2><p>Starting in Storybook 10, we're shipping ESM-only builds.</p><p>This change delivers several significant benefits:</p><ul><li>Smaller package size (15%)</li><li>Cleaner codebase with fewer conditionals and workarounds</li><li>Reduced bugs from module duplication and aliasing</li><li>A more future-proof foundation for Storybook and its ecosystem</li></ul><p>We've worked hard to maintain backward compatibility wherever possible. Most addons should continue to work seamlessly. For those that don't, we'll provide clear guidance and support.</p><p>You should be aware of these breaking changes:</p><ul><li>You'll need a Node.js version that supports <code>require()</code> of ESM, these are <code>20.16</code>, <code>22.19</code>, <code>24</code> or higher.</li><li>Your Storybook config and story files must be valid ESM</li><li>More details available in our <a href="https://storybook.js.org/docs/10/releases/migration-guide?ref=storybookblog.ghost.io">migration guide</a></li></ul><h2 id="%F0%9F%A7%AA-try-storybook-10-today">🧪 Try Storybook 10 today</h2><p>Storybook 10 is currently in beta, with the stable release planned for a few weeks later.</p><p>You can try it in a new project with this command:</p><pre><code class="language-bash">npm create storybook@next
</code></pre><p>If you have an existing Storybook 9 project, you can upgrade with this command. Use our automated migration wizard to help you along the way:</p><pre><code class="language-bash">npx storybook@next upgrade
</code></pre><p>You can review the changes in our <a href="https://storybook.js.org/docs/10/releases/migration-guide?ref=storybookblog.ghost.io">migration guide</a>.</p><h2 id="%F0%9F%A6%B8-looking-forward">🦸&nbsp;Looking forward</h2><p>ESM is a significant step toward a healthier, simpler, and more unified JavaScript ecosystem. While dual module support served its purpose in getting us to this point, it's now time to move forward.</p><p>We're excited to make this transition in Storybook 10 and can't wait to see what new possibilities it unlocks.</p><hr /><p>We’d like to thank the following pioneers that made this change possible:</p><ul><li><strong>Titus Wormer</strong> (<code>@wooorm</code>) - Pushed the entire unified ecosystem to ESM despite community pushback and wrote extensively about the decision</li><li><strong>Rich Harris</strong> - Creator of Svelte and early ESM advocate</li><li><strong>Marvin Hagemeister</strong> - Maintainer of Preact who helped navigate its transition</li><li><strong>Fred K.
